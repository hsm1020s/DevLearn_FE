### 2-3. 인증 실패 시 응답 포맷 불일치

| 상황 | BE 실제 응답 | FE 기대 | 영향 |
|------|--------------|---------|------|
| 토큰 없이 보호된 경로 | **403 Forbidden** + `{timestamp,status,error,path}` (ApiResponse 미적용) | 401 (refresh 트리거) | FE 의 자동 토큰 갱신이 **절대 작동하지 않음** — [src/services/api.js:31](src/services/api.js#L31) 은 `status === 401` 조건만 감지 |

## 3. FE ↔ BE 계약 정적 diff

라이브 테스트 외에 FE 가 기대하는 응답 shape 과 BE DTO 불일치 지점.

| # | 지점 | FE 기대 | BE 실제 | 영향 |
|---|------|---------|---------|------|
| C1 | `chatApi.updateConversation` → `.data.data` | 업데이트된 metadata 객체 | `ApiResponse<Void>` (data=null) | `useChatStore.renameConversation` 등에서 반환값 안 씀 → 현재는 무해, 향후 리팩토링 시 주의 |
| C2 | `chatApi.deleteConversations` | `{deletedIds: string[]}` | `ApiResponse<Void>` | 현재 fire-and-forget 이라 무해, JSDoc 타입 수정 필요 |
| C3 | 로그인 응답 `user` | FE JSDoc 에 명시 없음 | `{name, email, role}` — **`id` 누락** | `useAuthStore` 가 user.id 를 사용하면 undefined. 현재 코드에서 id 참조는 없음(안전). |
| C4 | FE 500 인터셉터 | `status >= 500 || status === 404` → `/error/{status}` 강제 이동 | BE는 거의 모든 에러를 500으로 반환 (B1~B9) | **통합 환경에서 어떤 CRUD 액션이든 에러 발생 시 즉시 에러 페이지로 이동** — 앱이 사실상 사용 불가 |
| C5 | FE `errorHandler` | `error.response.data.message` 를 토스트로 표시 | ApiResponse 형식이면 OK, 403 은 `error` 필드만 있음 | 403 시 "요청에 실패했습니다" 로 표시되나 에러 페이지로 리다이렉트는 안 함(500/404 조건에 안 걸림) → 정적 에러 안내 불가 |

## 6. 결론

- FE 통합 계약 자체는 대체로 정렬되어 있음 (`/api/**` 경로, ApiResponse 래핑 해제 로직, CORS 설정 전부 호환).
- 그러나 **BE가 거의 모든 변경/조회-상세 엔드포인트에서 500을 반환**하고 있어, 통합 환경에서는 앱이 사실상 동작하지 않음.
- FE는 500/404 를 글로벌 에러 페이지로 강제 이동시키는 인터셉터가 있어, BE 500 폭발과 맞물려 **사용자가 어떤 조작을 해도 즉시 에러 페이지로 튕기는 상태**.
- 인증 실패 시 401이 아닌 403 반환은 FE의 자동 토큰 갱신 로직을 무력화 — 15분 이후 모든 호출이 에러 페이지로 이동.

우선 P0 3건(B1~B9 BE 500, C4 FE 과잉 리다이렉트, 인증 401/403 불일치)을 해결해야 MVP 수준의 풀스택 동작이 가능.
